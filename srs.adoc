== Introduzione

=== Scopo del Documento

Il presente documento costituisce la *Specifica dei Requisiti del Software (SRS)* per l'applicazione *StudyTogether*, sviluppata con la piattaforma https://studio.code.org/projects/applab/[AppLab] di https://code.org[Code.org].
La SRS definisce in modo formale e completo le funzionalità, i vincoli e i comportamenti attesi dell'applicazione, servendo come riferimento per il team di sviluppo.

Lo standard di riferimento adottato è **ISO/IEC/IEEE 29148:2018** — _Systems and software engineering — Requirements engineering_.

=== Scopo dell'Applicazione e Visione Collaborativa

StudyTogether nasce dall'idea che lo studio non debba essere un atto isolato e solitario, bensì un *flusso di lavoro condiviso e comunitario*.
L'informatica è qui intesa non come fine, ma come *mezzo di collaborazione e cooperazione tra pari*.

L'app trasforma la classe in una **Comunità di Pratica** (Lave & Wenger, 1991): uno spazio in cui la conoscenza non è detenuta da pochi, ma circola, si costruisce collettivamente e si consolida attraverso la partecipazione attiva di ogni studente.

Le interrogazioni programmate, tradizionalmente vissute come momenti di ansia e competizione, diventano *occasioni di apprendimento cooperativo*: gli studenti gestiscono insieme il calendario delle verifiche, si candidano consapevolmente, condividono quesiti di studio e organizzano la conoscenza per argomento e materia.

=== Ambito di Applicazione

L'applicazione è destinata agli studenti delle classi della scuola secondaria di secondo grado.
Il contesto d'uso è quello della *singola classe scolastica* che adotta l'app come strumento condiviso di autogestione dello studio.
La piattaforma AppLab di Code.org consente lo sviluppo di app data-driven per smartphone con un database integrato, rendendola adatta a questo caso d'uso didattico.

== Descrizione Generale del Sistema

=== Prospettiva del Sistema

StudyTogether è un'applicazione per smartphone, sviluppata interamente su AppLab (Code.org).
Utilizza il database integrato della piattaforma per la persistenza dei dati.
Non richiede connessione a server esterni né autenticazione con account personali: il modello è quello di un'app *condivisa dalla classe*, installata (o accessibile via browser) su dispositivi mobili individuali. Sta alla responsabilità degli studenti agire in modo etico.

=== Funzionalità Principali

Le funzionalità principali consistono nell'inizializzazione dell'applicazione (inserimento di: anagrafiche degli studenti, materie, argomenti, quesiti, materiali didattici), nella gestione delle interrogazioni programmate (creazione, visualizzazione, candidatura), nella gestione delle interrogazioni svolte (tipo di candidato, studenti interrogati, inserimento nuovi quesiti) e nella visualizzazione del calendario condiviso. In sintesi, le funzionalità chiave sono:

* Gestione anagrafica degli studenti della classe
* Catalogazione delle materie per le quali sono previste interrogazioni
* Organizzazione degli argomenti per materia
* Archiviazione dei quesiti di studio per argomento
* Archiviazione dei materiali didattici
* Archiviazione dei riferimenti specifici che legano i quesiti ai materiali didattici
* Pianificazione delle interrogazioni programmate con data, ora e numero di posti
* Registrazione delle interrogazioni svolte con indicazione degli studenti effettivamente interrogati
* Visualizzazione del calendario delle interrogazioni
* Candidatura spontanea degli studenti alle interrogazioni programmate
* Visualizzazione dei candidati per ogni interrogazione programmata o svolta
* Visualizzazione degli studenti in attesa di interrogazione su un argomento

=== Caratteristiche degli Utenti

L'unico profilo utente è quello dello **Studente**.
Non è prevista una distinzione tra amministratori e utenti ordinari: la filosofia cooperativa dell'app prevede che tutti i membri della classe abbiano gli stessi diritti e responsabilità.
Ogni studente può inserire, visualizzare e candidarsi liberamente.

=== Vincoli e Assunzioni

* Sviluppo esclusivo su piattaforma AppLab di Code.org.
* Utilizzo del database integrato di AppLab (API: https://studio.code.org/docs/ide/applab/expressions/createRecord[`createRecord`], https://studio.code.org/docs/ide/applab/expressions/readRecords[`readRecords`], https://studio.code.org/docs/ide/applab/expressions/updateRecord[`updateRecord`], https://studio.code.org/docs/ide/applab/expressions/deleteRecord[`deleteRecord`]).
* Interfaccia ottimizzata per smartphone (schermo verticale, touch-first, dimensioni 320x450 px).
* Nessun sistema di autenticazione individuale: responsabilità condivisa dei dati.
* Lingua dell'interfaccia: Italiano.
* L'app è progettata per essere usata da una singola classe per volta.

```js
function violaUnicita(tupla, relazione, attributi) {
  // relazione deve essere una lista
  // attributi è un solo attributo
  if (typeof attributi === 'string') {
    // violazione se esiste un elemento con stesso attributo
    for (var i = 0; i < relazione.length; i++) {
      var tuplaCorrente = relazione[i];
      if (tuplaCorrente[attributi] == tupla[attributi]) {
        console.log("Vincolo di unicità (singolo attributo) non rispettato");
        return true;
      }
    }
    return false;
  } else {
    // violazione se esiste un elemento con tutti gli attributi uguali
    for (i = 0; i < relazione.length; i++) {
      tuplaCorrente = relazione[i];
      var violazione = true;
      for (var j = 0; j < attributi.length; j++) {
        if (tuplaCorrente[attributi[j]] != tupla[attributi[j]])
          violazione = false;
      }
      if (violazione) {
        console.log("Vincolo di unicità (attributi multipli) non rispettato");
        return true;
      }
    }
    return false;
  }
}
onEvent("btnSalvaStudente", "click", function( ) {
  // posizione nel registro elettronico
	var posizione = parseInt(getProperty("txtPosizione", "text"));
	var cognome = getProperty("txtCognome", "text");
	var nome = getProperty("txtNome", "text");
	// crea l'oggetto studente
	var studente = {
	  "posizione": posizione,
	  "cognome": cognome,
	  "nome": nome
	};
	var elenco = readRecordsSync("Studente");
	if (!violaUnicita(studente, elenco, "posizione") &&
	    !violaUnicita(studente, elenco, ["cognome", "nome"]))
  	createRecord("Studente", studente,

  	});
});
```

== Requisiti Non Funzionali

=== Usabilità

* L'interfaccia deve essere utilizzabile senza formazione preliminare da studenti di scuola secondaria.
* La navigazione principale deve avvenire tramite barra di navigazione bottom-tab con massimo 5 voci.
* Ogni operazione deve essere completabile in massimo *3 tap* dalla schermata principale.
* I messaggi di errore devono essere scritti in linguaggio naturale, non tecnico.

=== Affidabilità

* Il sistema deve validare tutti i dati in input prima del salvataggio.
* In caso di errore di salvataggio, deve essere mostrato un feedback chiaro allo studente.
* Non devono essere possibili duplicati nei campi chiave.

=== Prestazioni

* Il caricamento di qualsiasi schermata non deve superare i *3 secondi* su connessione standard.
* Il database è dimensionato per: ≤ 30 studenti, ≤ 20 materie, ≤ 200 argomenti, ≤ 500 quesiti, ≤ 100 interrogazioni.

=== Manutenibilità

* Il codice deve essere commentato e strutturato in funzioni con responsabilità singola.
* Le operazioni di database devono essere incapsulate in funzioni helper riutilizzabili.

=== Vincoli di Progettazione (AppLab)

* Tutte le schermate devono essere progettate nel _Design Mode_ di AppLab.
* La logica applicativa deve essere scritta in JavaScript ES6 compatibile con l'interprete di AppLab.
* Le operazioni di database devono usare esclusivamente le API native di AppLab:
+
[source,javascript]
----
createRecord(tableName, record, callback)
readRecords(tableName, callback)
updateRecord(tableName, record, callback)
deleteRecord(tableName, record, callback)
----
* Non è possibile usare librerie esterne non supportate da AppLab.

== Visione Pedagogica e Collaborativa

=== L'Informatica come Strumento di Cooperazione

StudyTogether non è semplicemente un'agenda digitale: è uno strumento che reifica il principio secondo cui *imparare è un atto sociale*.

La tecnologia, in questo contesto, non automatizza né sostituisce la relazione educativa: la *amplifica*.
Ogni studente che inserisce un quesito arricchisce la base di conoscenza comune.
Ogni candidatura è un atto di responsabilità verso la classe.
Il calendario condiviso trasparente riduce l'ansia da prestazione e aumenta la prevedibilità del percorso di studio.

Il modello si ispira al *cooperative learning* (Johnson & Johnson, 1989): interdipendenza positiva, responsabilità individuale, interazione faccia a faccia, competenze sociali e revisione di gruppo si traducono in funzionalità concrete dell'app.

=== Mapping Pedagogia → Funzionalità

[cols="2,3", options="header"]
|===
| Principio Pedagogico | Funzionalità StudyTogether

| Interdipendenza positiva
| Il calendario è condiviso: nessuno può ignorare le interrogazioni degli altri.

| Responsabilità individuale
| La candidatura è un atto volontario e tracciato.

| Costruzione collettiva della conoscenza
| I quesiti inseriti da chiunque diventano patrimonio di tutti.

| Metacognizione
| Organizzare argomenti e quesiti stimola la riflessione sul sapere.

| Riduzione dell'ansia da prestazione
| La visibilità dei posti e delle date riduce l'incertezza.

| Autoregolazione dell'apprendimento
| Lo studente sceglie quando candidarsi, assumendosi la responsabilità.
|===
